# Чат-бот отдела продаж (MVP)

Telegram-бот первичной поддержки/продаж для B2B SaaS-продукта. Отвечает на
типовые вопросы по базе знаний через GigaChat (ProxyAPI), квалифицирует лиды
с явного согласия клиента и передаёт заявки в CRM-заглушку. Полная
спецификация — в файле `Спецификация MVP чат-бота.md`.

## Стек

Python 3.11+, aiogram 3 (Telegram), httpx + tenacity (GigaChat через
ProxyAPI, retry с экспоненциальной задержкой), pydantic / pydantic-settings,
SQLite (статистика и лиды).

## Структура проекта

```
bot/
  config.py           — настройки из .env
  models.py            — Lead, DialogSession и связанные модели
  knowledge_base.py    — база знаний (ЗАГЛУШКА — заменить на данные заказчика)
  gigachat_client.py   — обёртка над GigaChat/ProxyAPI: retry, валидация, mock-режим
  crm_stub.py           — заглушка REST API для лидов (POST /leads), SQLite + опциональный вебхук
  stats.py               — статистика взаимодействий в SQLite
  dialog.py              — движок диалога (не зависит от Telegram)
  handlers.py            — aiogram-обработчики (тонкий транспортный слой)
  main.py                 — точка входа Telegram-бота
console_test.py         — консольный тест-харнесс (без Telegram и без ключей)
tests/                    — unit-тесты (gigachat_client, crm_stub, dialog)
```

## Установка

```bash
python -m venv .venv
.venv\Scripts\activate        # Windows
pip install -r requirements-dev.txt
copy .env.example .env        # заполнить реальными ключами, когда они появятся
```

## Как подключён GigaChat (через ProxyAPI)

Бот никогда не вызывает GigaChat напрямую из Telegram-обработчиков — весь
путь одного вопроса идёт так:

1. `bot/dialog.py` классифицирует тему вопроса по ключевым словам
   (`bot/knowledge_base.py: find_topic`) и берёт готовый факт-контекст из
   базы знаний.
2. Контекст подставляется в системный промпт (`dialog.py:
   SYSTEM_PROMPT_TEMPLATE`), который жёстко запрещает модели отвечать
   что-либо за пределами переданного контекста.
3. `bot/gigachat_client.py: GigaChatClient.generate()` — тонкая httpx-обёртка
   над ProxyAPI: `POST {PROXYAPI_BASE_URL}` с заголовком
   `Authorization: Bearer {PROXYAPI_KEY}`, тело в OpenAI-совместимом формате
   (`{model, messages: [...]}`). Сетевые сбои, 429 и 5xx уходят в retry с
   экспоненциальной задержкой (`tenacity`, `RETRY_ATTEMPTS` попыток), после
   исчерпания retry наружу — единообразная `GigaChatUnavailableError`.
4. Если GigaChat/ProxyAPI недоступны даже после retry — пользователь не
   видит техническую ошибку: `dialog.py` честно передаёт вопрос менеджеру
   (создаёт лид со статусом `needs_manager`), как того требует
   спецификация.

**Текущий статус:** `PROXYAPI_KEY` не задан ни у заказчика, ни в этом
проекте, поэтому боевого обращения к GigaChat ещё не было ни разу. Без
ключа `GigaChatClient` автоматически заменяется на `MockGigaChatClient`
(`bot/gigachat_client.py: get_gigachat_client`) — офлайн-заглушку, которая
отвечает напрямую текстом из базы знаний, без сети. Вся остальная логика
(классификация темы, retry-контур, эскалация на менеджера) при этом
работает и протестирована так же, как будет работать с реальным ключом.

**Чтобы включить реальный GigaChat:** вписать `PROXYAPI_KEY` (и при
необходимости `GIGACHAT_MODEL`) в `.env` — переменные окружения на
хостинге (например, `railway variable set PROXYAPI_KEY=...`). Код менять
не нужно. Перед первым боевым вызовом стоит сверить `PROXYAPI_BASE_URL` и
формат тела запроса с актуальной документацией ProxyAPI — сейчас
предполагается OpenAI-совместимая схема, в коде это явно помечено как
`ЗАГЛУШКА/ПРОВЕРИТЬ`.

## Тестирование без ключей (доступно уже сейчас)

Ключей GigaChat/ProxyAPI/Telegram пока нет — но всю логику диалога можно
проверить локально, без сети: без `PROXYAPI_KEY` бот автоматически переходит
в офлайн-режим (`MockGigaChatClient`) и отвечает напрямую на основе базы
знаний.

```bash
python console_test.py
```

Пример сценария для проверки:

1. `Сколько стоит подписка?` — бот ответит по базе знаний и (при явном
   покупательском интересе) спросит согласие на передачу контакта менеджеру.
2. `да` → бот запросит имя, контакт и комментарий.
3. После сбора данных — заявка сохраняется в `data/leads.sqlite3`.
4. `какая сегодня погода` — бот вежливо ограничит тему разговора продуктом.
5. `пока` → бот попросит оценку (👍/👎) и завершит диалог.

Проверить, что записалось, можно так:

```bash
python console_test.py --show-leads
python console_test.py --show-stats
```

## Unit-тесты

```bash
pytest
```

Тесты мокают сетевые вызовы (GigaChat/ProxyAPI и CRM-вебхук) через
`unittest.mock`, реальная сеть не используется.

## Запуск реального Telegram-бота

Когда появятся ключи — заполните `.env`:

- `TELEGRAM_BOT_TOKEN` — токен бота от @BotFather
- `PROXYAPI_KEY` — ключ доступа ProxyAPI к GigaChat
- при необходимости `PROXYAPI_BASE_URL` — уточните точный путь эндпоинта в
  актуальной документации ProxyAPI (в коде явно помечено как проверить)
- `CRM_WEBHOOK_URL` — опционально, реальный REST-эндпоинт приёма лидов;
  без него заявки сохраняются только локально в SQLite

```bash
python -m bot.main
```

## Ограничения и правила поведения (реализовано)

- Бот отвечает только на основе базы знаний — не придумывает цены, сроки и
  гарантии (`bot/dialog.py: SYSTEM_PROMPT_TEMPLATE`, `bot/knowledge_base.py`).
- Контактные данные запрашиваются и сохраняются только после явного
  согласия клиента (`bot/dialog.py: _handle_consent_answer`).
- Вопросы вне базы знаний или полная недоступность GigaChat приводят к
  честному ответу «передам менеджеру» и созданию заявки в CRM-заглушке, а
  не к угадыванию ответа.
- Вопросы не по теме продукта вежливо ограничиваются рамками сценария, без
  обращения к нейросети.
- Ключи хранятся только в `.env`, не логируются и не передаются в промпт.

## Известные ограничения MVP

- Сессии диалога хранятся в памяти процесса (`bot/handlers.py`) — при
  перезапуске бота диалог с клиентом начинается заново. Статистика и лиды
  сохраняются в SQLite и не теряются.
- Классификация темы вопроса — rule-based по ключевым словам
  (`bot/knowledge_base.py: find_topic`), без дополнительного ML-слоя —
  достаточно для 8 типовых тем MVP, при расширении FAQ стоит пересмотреть.
- GigaChat пока не подключён боевым ключом — подробности и что нужно для
  включения см. «Как подключён GigaChat» выше.
