# Спецификация MVP чат-бота для поддержки отдела продаж

**Проект:** Чат-бот первичной коммуникации с клиентами (заказ с фриланс-маркетплейса)
**Стек:** GigaChat API (через ProxyAPI), Telegram, Python
**Дата:** 15 июля 2026

---

## Заказчик и задача

Заказ с фриланс-маркетплейса — чат-бот первичной поддержки/продаж для B2B SaaS-продукта с подпиской. Задача: отвечать потенциальным клиентам (лидам) в Telegram на вопросы о продукте, квалифицировать лид и передавать сложные обращения менеджеру. Реальных CRM/API у заказчика нет — используется тестовая заглушка, которую позже можно заменить на настоящую интеграцию.

## Основные задачи бота

1. Отвечать на частые вопросы о продукте (цена, подписка, интеграции, поддержка) на основе базы знаний.
2. Квалифицировать лид — с явного согласия клиента собрать контакт и суть запроса. Без согласия контактные данные не запрашиваются и не сохраняются.
3. Передавать данные о лиде во внешнюю систему (пока — тестовая заглушка REST API).
4. При нехватке данных или сложном вопросе — переводить диалог на менеджера, не придумывая ответ.
5. Собирать статистику взаимодействий (тема вопроса, время ответа, удовлетворённость) для анализа эффективности.

## Типовые вопросы клиентов (5–8)

1. Сколько стоит подписка и какие есть тарифы?
2. Какие условия подписки — можно ли отменить, есть ли пробный период?
3. Какие интеграции поддерживает продукт (CRM, API, другие сервисы)?
4. Как связаться с поддержкой и сколько занимает ответ?
5. Чем отличаются тарифные планы между собой?
6. Есть ли демо-доступ или бесплатный тест?
7. Какие способы оплаты доступны?
8. Что делать, если возникла техническая проблема?

## Целевая аудитория

B2B — специалисты и компании, знакомые с техническими терминами (API, интеграция). В основном новые лиды на этапе принятия решения о покупке, ещё не клиенты. Тон бота: профессиональный, по делу, без объяснения базовых понятий, с акцентом на факты и следующий шаг (заявка/консультация).

## Платформа и причина выбора

**Telegram** — выбор заказчика. Вероятная причина (не подтверждена явно заказчиком, но логична для B2B-аудитории): широкое использование Telegram специалистами и компаниями в СНГ, удобный и хорошо документированный Bot API, более быстрая разработка по сравнению с ВКонтакте для этого сегмента.

## Интеграции

| Компонент | Решение | Обоснование |
|---|---|---|
| Нейросеть | GigaChat через ProxyAPI (SDK `gigachat` или `httpx`) | Официальный SDK управляет авторизацией, retry, поддерживает async и потоковую генерацию ([Sber Developers](https://developers.sber.ru/docs/ru/gigachat/guides/using-sdks)) |
| Платформа | Telegram Bot API через `aiogram` | Современный, асинхронный, активно поддерживаемый фреймворк |
| Внешняя система (CRM) | Тестовая заглушка REST API | Реальной системы нет; структура лида готова к замене на настоящий эндпоинт |
| Хранилище статистики | SQLite | Легко разворачивается вместе с ботом, доступно для SQL-анализа |

**Структура заглушки лида:**
```json
{
  "lead_id": "uuid",
  "name": "строка",
  "contact": "телефон/telegram_id",
  "question_topic": "цена | подписка | интеграции | поддержка",
  "status": "new | needs_manager",
  "created_at": "ISO-timestamp"
}
```

## База знаний для бота (услуги, цены, FAQ)

Поскольку реальных данных нет, создаётся демо-база с явно помеченными тестовыми значениями:
- Тарифы (заглушка — Basic/Pro/Enterprise с примерными ценами)
- Условия подписки (заглушка)
- Список интеграций (заглушка)
- Контакты поддержки (заглушка)

Структура должна быть готова для лёгкой замены плейсхолдеров на реальные данные заказчика в будущем. Все тестовые значения должны быть явно помечены в коде как "ЗАГЛУШКА — заменить на реальные данные заказчика".

## Ограничения и правила поведения

- Не выдумывать факты, цены, сроки и гарантии сверх того, что есть в базе знаний.
- При отсутствии ответа в базе знаний — не угадывать, а переводить на менеджера.
- Ключи (GigaChat/ProxyAPI, Telegram Bot Token) хранятся в переменных окружения (`.env`), не передаются в промпт нейросети и не логируются.
- Тон общения — деловой, без панибратства, соответствующий B2B-аудитории.
- Ограничение темы диалога рамками продукта/продаж — не отвечать на несвязанные вопросы.
- Собирать контактные данные (имя, телефон, telegram-контакт) только после явного согласия клиента — например, через прямой вопрос «Хотите, чтобы менеджер связался с вами? Могу передать ему ваш контакт». Без согласия — не запрашивать и не сохранять контакт, ограничиться ответом по теме.

## Обработка ошибок и нестабильности

| Тип сбоя | Обработка |
|---|---|
| Сетевые сбои/таймауты GigaChat/ProxyAPI | Таймаут запроса (15–20 сек), retry с экспоненциальной задержкой (`tenacity`, 2–3 попытки) |
| Лимиты/авторизация (429, протухший токен) | Автообновление токена, пауза и повтор при превышении лимита |
| Невалидный/пустой ответ модели | Валидация перед отправкой пользователю, заготовленная безопасная фраза при сбое |
| Полная недоступность нейросети | Fallback-сообщение + передача диалога менеджеру, логирование инцидента |
| Сбои платформы (Telegram API) | Очередь отправки сообщений с повтором |

## Промпт для нейросети-разработчика (ClaudeCode)

```
Разработай MVP Telegram-бота для первичной поддержки продаж B2B SaaS-продукта на Python.

СТЕК:
- aiogram (последняя стабильная версия) — Telegram Bot API, async
- gigachat SDK или httpx — вызовы к GigaChat через ProxyAPI
- pydantic — валидация запросов/ответов и конфигурации
- python-dotenv — хранение ключей (GIGACHAT_CREDENTIALS, PROXYAPI_KEY, TELEGRAM_BOT_TOKEN) в .env
- tenacity — retry с экспоненциальной задержкой при сетевых сбоях/лимитах (2-3 попытки)
- sqlite3 (stdlib) или SQLAlchemy — хранение статистики
- logging (stdlib) — логирование ошибок и диалогов

СТРУКТУРА ПРОЕКТА:
/bot
  main.py            — точка входа, инициализация aiogram
  handlers.py        — обработчики сообщений и команд
  gigachat_client.py — обёртка над GigaChat/ProxyAPI с retry и валидацией ответа
  knowledge_base.py  — база знаний (FAQ, тарифы, условия) — все значения помечены как ТЕСТОВЫЕ/ЗАГЛУШКА
  crm_stub.py        — тестовая заглушка REST API для передачи лидов (POST /leads)
  stats.py           — запись статистики (тема вопроса, время ответа, оценка) в SQLite
  config.py          — загрузка .env, Pydantic-настройки
.env.example
requirements.txt

ФУНКЦИОНАЛЬНОСТЬ:
1. При старте диалога — приветствие, короткое объяснение возможностей бота.
2. Обработка вопроса пользователя: классификация темы (цена/подписка/интеграции/поддержка/другое) → поиск ответа в базе знаний → генерация ответа через GigaChat с системным промптом, ограничивающим модель отвечать ТОЛЬКО на основе переданного контекста базы знаний.
3. Если в базе знаний нет данных для ответа — НЕ передавать вопрос в нейросеть как открытый, а сразу вернуть сообщение о передаче менеджеру и создать заявку через crm_stub.py.
4. После 3-5 сообщений или явного намерения (запрос цены, готовность купить) — сначала явно спросить согласие клиента на передачу контакта менеджеру («Хотите, чтобы менеджер связался с вами?»). Только при согласии — запросить контакт и передать данные в crm_stub.py. Без согласия — не сохранять и не запрашивать контактные данные, продолжить отвечать по теме.
5. После завершения диалога — короткий запрос оценки (👍/👎), запись в stats.py вместе с темой вопроса и временем ответа.
6. Все ошибки API (таймауты, 429, невалидный ответ) — обрабатывать через retry, при исчерпании попыток — fallback-сообщение + передача менеджеру, без вывода технических деталей пользователю.

ОГРАНИЧЕНИЯ (обязательно закрепить в системном промпте нейросети):
- Не придумывать цены, сроки, гарантии и факты, отсутствующие в переданной базе знаний.
- При неуверенности — явно предлагать связаться с менеджером, а не гадать.
- Никогда не включать в промпт нейросети содержимое .env/ключей API.
- Запрашивать и сохранять контактные данные пользователя (имя, телефон, telegram-контакт) только после явного согласия. Без согласия — не собирать и не хранить контакт.

Добавь unit-тесты для crm_stub.py и gigachat_client.py (мок ответов). Все тестовые/демо-данные в base_knowledge.py должны быть явно закомментированы как "ЗАГЛУШКА — заменить на реальные данные заказчика".
```

## Два варианта поведения при нехватке данных

**Слабый вариант (не делать так):**
Бот сам придумывает правдоподобный ответ — например, называет примерную цену "около 5000 рублей в месяц" или сроки "обычно 2-3 дня", хотя точных данных у него нет. Это создаёт риск обмана клиента, юридические и репутационные проблемы для заказчика.

**Правильный вариант:**
Бот честно признаёт границу своих знаний — "Точных данных по этому вопросу у меня нет, чтобы не давать вам неверную информацию, я передам ваш вопрос менеджеру — он свяжется с вами в течение [время]" — и одновременно фиксирует лид в системе (даже в заглушке), чтобы обращение не потерялось.
